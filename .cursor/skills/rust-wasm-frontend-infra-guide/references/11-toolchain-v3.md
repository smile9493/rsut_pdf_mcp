# Toolchain & Lifecycle Automation Specification V3.2 (Ultimate Edition)

This specification is the engineering guarantee of the Rust+Wasm frontend infrastructure's "materialist foundation." By transforming physical protocols (layout, size, latency) from human discipline into an automated pipeline, it ensures architectural stability and performance determinism throughout the project lifecycle.

## 1. Physical Layout Automation & Synchronization

### 1.1 Compile-Time Physical Layout Assertions (MUST)

All `#[repr(C)]` structures that interact via `WasmSlice` or shared memory must harden their layout at compile time using `assert_layout` or static assertion macros.

**Mandatory Verification Items**:
- Total struct size (`SizeOf`)
- Key field offsets (`OffsetOf`)

**Physical Significance**: Eliminates memory alignment/reordering changes caused by Rust compiler version updates or code refactoring, preventing runtime crashes from JS-side index failures.

```rust
// wasm-kernel/src/infra/layout.rs
#[repr(C)]
pub struct DrawCommand {
    pub op: u8,        // Offset 0
    pub _pad: [u8; 3], // Explicit alignment padding
    pub x: f32,        // Offset 4
    pub y: f32,        // Offset 8
    pub color: u32,    // Offset 12
}

// Compile-time assertion interception: compilation fails if layout changes
#[cfg(test)]
mod tests {
    use super::*;
    use std::mem::{size_of, offset_of};

    #[test]
    fn test_draw_command_layout() {
        assert_eq!(size_of::<DrawCommand>(), 16);
        assert_eq!(offset_of!(DrawCommand, x), 4);
        assert_eq!(offset_of!(DrawCommand, y), 8);
        assert_eq!(offset_of!(DrawCommand, color), 12);
    }
}
```

### 1.2 TypeScript Type Contract Generation (SHOULD)

For cross-language shared DTOs or instruction structures, adopt a progressive automation path:

1. **Small/Medium Scale**: Compile-time assertions + handwritten `.d.ts` completion.
2. **Large Scale / Complex Structures**: Introduce `tsify` or `wasm-bindgen-utils` to automatically export TS interface definitions.
3. **Iron Rule**: JS-side `DataView` offset parameters **must** reference constants generated by automation. Hardcoding `view.setFloat32(off + 4, ...)` is strictly forbidden.

```typescript
// ✅ Good — offset constants generated from Rust layout
import { DRAW_CMD_X_OFFSET, DRAW_CMD_Y_OFFSET } from './wasm-layout-constants';
view.setFloat32(DRAW_CMD_X_OFFSET, x, true);

// ❌ Bad — hardcoded offset
view.setFloat32(4, x, true);
```

## 2. Binary Footprint Audit Pipeline

### 2.1 Physical Size Budget System (Size Budget)

Establish a `.wasm-size-budget.json` based on business scenarios. Audit targets are strictly the final artifacts after `wasm-bindgen` CLI processing and `wasm-opt -Oz` optimization.

**Metrics**:
- `Raw .wasm Size`
- `Gzip Size`

**PR Interception Mechanism**: Execute in the CI pipeline's pre-merge stage. Exceeding the budget triggers manual review, accompanied by a `twiggy` diagnostic report.

```json
{
  "budget": {
    "raw_bytes": 200000,
    "gzip_bytes": 50000
  },
  "warning_threshold": 0.9
}
```

### 2.2 Space Cost Diagnostic Toolchain (Twiggy Integration)

Position `twiggy` as an auxiliary diagnostic tool.

**Analysis Dimensions**:
- `twiggy top`: Identify the top 20 largest code units.
- `twiggy dominators`: Trace generic expansion and dependency chains.

**Size Reduction Tactics**:
- **Limit `fmt` Usage**: Strictly prohibit `format!`, `to_string()`, or complex `Debug` derives in high-frequency tick paths or core logic.
- **Mandatory LTO**: Must enable `lto = true` and `codegen-units = 1`, relying on the linker for physical-level dead code elimination (DCE).

```toml
# Cargo.toml — profile.release
[profile.release]
lto = true
codegen-units = 1
opt-level = "z"
```

## 3. Physical Performance Sampling & Telemetry

### 3.1 Memory Actual Water Level Monitoring

Leverage physical information from the browser environment to establish real water-level telemetry:

**Physical Page Monitoring**: Sample `core::arch::wasm32::memory_size(0)` to obtain allocated pages.

**Fragmentation Rate Audit**: Record the delta between the Arena allocator's `allocated_bytes` and total memory pages, monitoring the physical utilization of linear memory.

### 3.2 Physical Link Latency Measurement (FFI Performance)

**Measurement Tool**: Mandatory use of microsecond-precision `performance.now()`.

**Metric Definition**: `Latency = T(Wasm_Tick_End) - T(JS_Flush_Start)`.

**Warning Red Line**: If single-frame latency fluctuation exceeds 4ms (25% of the 60fps budget), it is considered a physical regression.

```typescript
// JS-side latency measurement
const flushStart = performance.now();
engineFacade.flush();
// ... Wasm processes ...
// T(Wasm_Tick_End) reported via telemetry callback
const latency = wasmTickEnd - flushStart;
if (latency > 4.0) {
    console.warn(`[Telemetry] FFI latency regression: ${latency.toFixed(2)}ms`);
}
```

## 4. Lifecycle Checkpoint Integration

Mandatory physical audit checkpoints inserted into **Stage 2: Design Blueprint** of the project lifecycle:

| Checkpoint | Audit Content | Judgment Criteria |
|:---|:---|:---|
| **Allocator Selection** | `mini-alloc` vs `talc` | < 50KB business should prioritize `mini-alloc` |
| **Physical Capacity Forecast** | Single-frame instruction traffic estimation | Must ensure no frequent DataView expansion |
| **Layout Assertion Coverage** | 100% shared struct assertions | Eliminate any implicit cross-language layout assumptions |

## 5. Hard Constraints

| ID | Constraint | Rationale |
|----|-----------|-----------|
| **[F-12] No Unoptimized `.wasm` Measurement** | Must not measure the size of unoptimized intermediate `.wasm` files. | Only the final `wasm-opt -Oz` artifact reflects the real physical footprint |
| **[F-13] No Absolute Size Veto Without Review** | Must not use an absolute size threshold for one-vote rejection without PR manual confirmation. | Context matters: a 10% increase may be justified by a critical feature |
| **[F-14] No Implicit Heap Allocation in Hot Paths** | Strictly prohibit any formatting operation that relies on implicit heap allocation in high-frequency paths. | `format!` and `to_string()` trigger allocator calls, causing frame stutter |
| **[F-15] No Pointer Exposure Without Layout Assertions** | Must not expose pointers to JS `DataView` without physical layout assertions. | Layout drift between Rust and JS causes silent data corruption |
